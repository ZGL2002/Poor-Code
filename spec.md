# 工具系统 Spec

## 背景
PoorCode 当前仅支持纯文本对话，模型无法读取文件、搜索代码、执行命令或修改文件。
用户每次需要模型操作代码时，只能手动复制粘贴，效率低下。
需要给 PoorCode 装上工具系统，让模型能从聊天机器人变成真正能干活的 Agent。

当前已完成：
- 多协议 Provider 抽象层（Anthropic + OpenAI），支持流式对话
- 终端交互界面（TUI），支持多轮上下文
- YAML 配置管理（含环境变量引用）

## 目标
- 定义统一的 Tool 抽象接口，各工具实现该接口即可被模型调用
- 提供六个核心工具：读文件、写文件、改文件、执行命令、模式找文件、搜代码内容
- 工具通过注册中心统一管理，按名查找，转成各协议要求的格式
- LLM 客户端能解析流式工具调用（JSON 参数碎片拼接），执行后将结果回灌进对话历史
- 单次回复限定一个工具调用，多工具调用时报错提示模型修正
- 工具执行带超时和错误处理，失败信息返回给模型助其调整
- 文件操作默认限制在当前项目目录，命令执行不做限制（用户等价权限）
- 本次不做 Agent Loop（模型调用工具一次后停止，不回环继续调用）

## 功能需求

- F1: Tool 抽象接口
  - 每个工具实现统一接口，包含四个元信息（名称 `name`、描述 `description`、参数 Schema `parameters`）和一个异步执行方法 `execute`
  - 参数 Schema 用 Python dict 描述（JSON Schema 兼容），Provider 层各自转换
  - `execute` 方法接收字典参数，返回结构化结果（成功时含 `success=True`+数据，失败时含 `success=False`+错误描述）

- F2: 工具注册中心
  - 模块级注册表，提供 `register(name, tool_instance)` 和 `get(name)` 方法
  - 提供 `list_tools()` 返回全部已注册工具实例
  - 提供 `to_api_format(protocol)` 将全部工具描述转为指定协议的列表格式
  - 启动时自动注册全部六个工具

- F3: 六个核心工具
  - F3a: **Read**（读文件）——接收 `file_path`，返回文件文本内容和总行数。路径默认相对于工作目录。文件不存在或无权限时返回错误
  - F3b: **Write**（写文件）——接收 `file_path` 和 `content`，创建或覆盖文件。父目录不存在时自动创建。写入后返回文件路径和字节数
  - F3c: **Edit**（改文件）——接收 `file_path`、`old_string`、`new_string`。在文件中查找 `old_string` 精确唯一匹配并替换为 `new_string`。匹配不到时返回「未找到原文」，匹配到多处时返回「匹配到 N 处（不唯一）」并附带上下文，均不修改文件
  - F3d: **Bash**（执行命令）——接收 `command`，在子进程中执行并返回 stdout、stderr 和退出码。无路径限制
  - F3e: **Glob**（模式找文件）——接收 `pattern`，返回匹配的文件路径列表。支持 `**` 递归匹配。默认相对工作目录
  - F3f: **Grep**（搜代码内容）——接收 `pattern` 和可选的 `path`、`glob`。返回匹配行列表，每项含文件路径、行号、行内容。不限制结果数量上限

- F4: 流式工具调用解析
  - Anthropic 协议下解析 `content_block_start`（`tool_use`）、`content_block_delta`（`input_json_delta` 碎片拼接）、`content_block_stop` 事件流
  - OpenAI 协议下解析 `choices[0].delta.tool_calls` 流式碎片（`function.name` + `function.arguments` 拼接）
  - 两种协议统一产出 `ToolCallRequest` 结构（`tool_name: str`、`tool_input: dict`）
  - 流式期间识别到多个工具调用时，只执行第一个并产出错误提示

- F5: 工具执行与结果回灌
  - Chat Loop 检测到 `ToolCallRequest` 后：暂停流式渲染 → 展示工具调用状态（名称+结果状态）→ 执行工具 → 将结果构造成 `ToolResult` 结构
  - 工具结果以协议要求的消息格式追加到对话历史中（Anthropic：`role: user` + `tool_result` content block；OpenAI：`role: tool` + `tool_call_id`）
  - 模型收到工具结果后生成文本回复，流式展示，完成后停止（不进入下一轮工具调用）

- F6: 超时与错误处理
  - 所有工具默认执行超时 30 秒，特殊工具可覆盖（如 Bash 可设 120 秒）
  - 超时返回 `success=False` + 超时提示 + 建议（如缩小范围）
  - 工具执行异常（权限不足、文件不存在等）均返回结构化错误，不抛异常崩溃

- F7: 路径安全
  - Read/Write/Edit/Glob/Grep 的路径参数默认限制在当前工作目录内（解析后拒绝 `../` 越界路径）
  - Bash 不做路径限制
  - 路径检查可通过工具注册时的配置关闭

- F8: TUI 工具状态展示
  - 工具调用时，终端显示工具名称和执行状态（如「🔧 read … ⏳ 执行中」→「🔧 read … ✅ 完成」或「❌ 失败」）
  - 不展示工具返回的具体内容
  - 工具执行失败时，错误摘要展示在状态行，详细错误信息通过工具结果回传给模型

## 非功能需求

- N1: 可扩展性
  - 新增工具只需实现 Tool 接口并一行 `register()`，无需修改 Registry、Provider 或 Chat Loop
  - 新增 LLM 协议后端的工具格式转换在 Provider 子类中完成，不影响工具定义

- N2: 健壮性
  - 工具执行超时不阻塞对话——超时后返回错误结果，Chat Loop 继续正常运行
  - 流式工具调用参数 JSON 碎片格式异常时，给出明确错误提示而非静默丢弃
  - 工具返回内容过大（单次 >100KB）时自动截断并附加提示，防止撑爆上下文窗口
  - 子进程（Bash）崩溃时不影响主进程

- N3: 兼容性
  - 保持现有 Provider 接口 `chat(messages, system_prompt, stream)` 签名不变
  - 不配置工具时（纯文本对话），行为与当前版本完全一致
  - 兼容 Anthropic Messages API 和 OpenAI Chat Completions API 的工具调用格式

- N4: 可观测性
  - 工具执行失败时，终端状态行可见工具名+失败标记，不影响对话继续
  - 可通过日志（如设置环境变量 `POORCODE_LOG=debug`）输出工具调用的完整请求和结果，便于调试

## 不做的事

- 不做 Agent Loop（不自动循环调用工具，模型调用一次工具 → 得到结果 → 生成文本回复 → 停止）
- 不做多工具并行/串行调用（本次严格限定单次回复最多一次工具调用）
- 不做工具调用权限确认（用户无确认环节，模型直接触发执行）
- 不做历史持久化（工具调用历史和结果随会话退出而丢弃，与现有 F6 行为一致）
- 不做 MCP 协议/外部工具接入
- 不做工具调用的限速（rate limiting）
- 不做 Bash 命令白名单/黑名单过滤
- 不做图片/二进制文件读写（工具仅处理文本文件）
- 不做 tool use 的流式取消/中断（开始执行就必须跑完或超时）

## 验收标准

- AC1: 启动 PoorCode，输入「读一下 README.md」，模型调用 Read 工具 → 终端显示「🔧 read … ✅ 完成」→ 模型基于文件内容生成文本回复
- AC2: 输入「把 README.md 中的 PoorCode 全部替换为 RichCode」，模型调用 Edit 工具 → 精确匹配到一处并替换成功 → 终端显示「🔧 edit … ✅ 完成」→ 再用 Read 确认替换生效
- AC3: 输入「在 README.md 中找一个不存在的字符串 abcxyz123」，模型调用 Edit 工具 → 返回「未找到原文」→ 模型在文本回复中说明匹配失败并建议调整
- AC4: 输入「写一个新文件 test.txt，内容是 hello world」，模型调用 Write 工具 → 终端显示「🔧 write … ✅ 完成」→ 文件被创建，内容正确
- AC5: 输入「执行 ls -la 看看项目目录」，模型调用 Bash 工具 → 终端显示「🔧 bash … ✅ 完成」→ 模型列出目录内容
- AC6: 输入「找到项目中所有 .py 文件」，模型调用 Glob 工具 → 返回文件列表 → 模型列出 Python 文件
- AC7: 输入「搜索代码中所有 import 语句」，模型调用 Grep 工具 → 返回匹配行 → 模型汇总搜索结果
- AC8: 一次对话中使用两种不同协议（先 anthropic 后 openai，需改配置重启），工具调用均正常
- AC9: 模型在一次回复中尝试调用多个工具时，只执行第一个并返回错误提示，对话不崩溃
- AC10: 执行 `sleep 120` 命令（Bash 工具超时），30 秒后返回超时错误，对话继续可用
- AC11: 输入「读 /etc/passwd」，Read 工具拒绝越界路径，返回错误，模型说明无法访问
- AC12: 工具调用失败后，模型能基于错误信息调整策略（如文件不存在时换文件名重试）
- AC13: 不配置工具时（关闭工具注册），对话行为与当前版本完全一致
