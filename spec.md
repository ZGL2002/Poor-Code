# Agent Loop Spec

## 背景
当前 PoorCode 的工具系统是「一步一停」模式：用户提问 → 模型调用一个工具 → 执行 → 模型生成文本回复 → 停止。模型不会自主判断「任务还没做完，需要再调一个工具」。这导致用户必须持续催促，每次模型停下来了就要再推一把，无法实现真正的自主编程助手体验。

当前已完成：
- 多协议 Provider 抽象层（Anthropic + OpenAI），支持流式对话和 tool use
- 六核心工具系统（Read / Write / Edit / Bash / Glob / Grep）
- 工具注册中心，支持按名查找和协议格式转换
- 终端 TUI（Rich 渲染、流式展示、工具状态行）
- Chat Loop 编排（单次工具调用 + 结果回灌）

## 目标
- 实现 ReAct 模式 Agent Loop：模型思考 → 调用工具 → 获取结果 → 继续思考 → 直到任务完成，全程自主推进
- 定义一套丰富的异步事件流，让 Agent 核心逻辑和 TUI 界面彻底解耦
- 支持多工具调用时的安全性分批——读类工具并发跑，写类工具串行跑
- 提供 Plan Mode 两段式工作流——/plan 先调研出方案，/do 再动手执行
- 多种停止条件覆盖安全网：模型自然结束、迭代上限、用户取消、连续未知工具、流错误

## 功能需求

- F1: Agent Loop（ReAct 循环）
  - 用户输入后追加到对话历史
  - 调用 LLM，流式接收响应
  - 响应结束：若模型产出了工具调用则执行工具并将结果回灌历史，继续循环；否则循环结束
  - 多轮循环由 Agent 自主推进，无需用户每轮催促

- F2: 停止条件（按优先级排列）
  - F2a: 模型自然结束——LLM 响应不含任何工具调用，纯文本回复即停止
  - F2b: 迭代上限——循环达到默认 25 轮上限（通过 config.yaml 配置），停止并给出提示
  - F2c: 用户取消当前轮次——按 Esc，取消本轮 Agent Loop，回到输入等待状态
  - F2d: 用户退出程序——按 Ctrl+C，退出整个 PoorCode
  - F2e: 连续未知工具——连续 3 次调用未注册的工具名，停止并提示
  - F2f: 流错误——LLM 流式响应期间发生不可恢复错误，停止并展示错误

- F3: 异步事件流
  - 定义 AgentEvent 类型集合：text_delta、thinking_delta、tool_call、tool_result、token_usage、agent_progress、agent_done、error
  - Agent 模块通过 AsyncIterator[AgentEvent] 产出事件
  - TUI 消费事件并渲染，Agent 不直接操作 TUI
  - 事件流让 Agent 核心逻辑和界面彻底解耦

- F4: 流式收集器（双路）
  - 实时路：每个 text_delta 立即以事件推送，TUI 实时展示
  - 累积路：完整响应（文本 + 工具调用列表）在流结束后供 Agent Loop 判断是否有工具调用、是否需要继续循环

- F5: 多工具调用安全分批
  - 模型一次返回多个工具调用时，按副作用分类
  - 读类工具（Read、Glob、Grep）并发执行
  - 写类工具（Write、Edit、Bash）串行执行
  - 混合时：先并发跑所有读类，全部完成后再串行跑写类
  - 所有工具结果按调用顺序组装，追加到对话历史

- F6: Plan Mode（两段式工作流）
  - /plan 切换至计划模式，仅向 LLM 暴露读类工具（Read、Glob、Grep）
  - /do 切换至执行模式，恢复全部六个工具
  - 手动切换，切换后保持在该模式直到用户再次切换
  - 切换实质上是从工具注册表中筛选子集传给 Provider

- F7: 工具结果回灌
  - 一次 LLM 响应可能包含多个工具调用，结果打包追加到历史
  - 不同协议按各自格式构造（Anthropic：user 消息 + tool_result block；OpenAI：tool 消息）
  - 回灌后不丢失历史中已有的消息

- F8: Agent 模块
  - 新增独立的 Agent 模块，封装循环逻辑
  - 通过事件流与外部通信
  - Chat Loop 变为薄编排层，连接 Agent、Provider 和 TUI

## 非功能需求

- N1: 可扩展性——新增停止条件只需在 Agent Loop 中添加一个检查函数；新增 AgentEvent 类型不影响已有事件的消费逻辑
- N2: 健壮性——单个工具执行失败不中断 Agent Loop，错误结果回灌给模型助其调整；Bash 子进程崩溃不影响主进程；循环中断后对话历史保留，用户可继续手动交互
- N3: 兼容性——保持 Provider 接口 `chat(messages, system_prompt, stream)` 签名不变；不传工具时行为与当前版本完全一致；现有六个工具的接口不受影响
- N4: 可观测性——每个工具调用的执行状态实时推送到 TUI；agent_progress 事件展示当前进度（如「第 3/25 轮」）；可通过环境变量 POORCODE_LOG=debug 输出完整 Agent Loop 日志
- N5: 配置化——迭代上限通过 config.yaml 配置（默认 25）；同一配置文件不引入破坏性变更

## 不做的事

- 不做权限系统（工具调用无需用户确认，模型直接触发执行）
- 不做上下文压缩（对话历史持续增长，超出模型上下文窗口时的处理留给后续章节）
- 不做用户交互式确认（不弹出「是否允许执行工具 X？」的确认环节）
- 不做 MCP 协议/外部工具接入
- 不做历史持久化（工具调用历史和结果随会话退出丢弃）
- 不做工具调用的流式取消/中断（工具开始执行就必须跑完或超时）
- 不做限速（rate limiting）

## 验收标准

- AC1: 输入「读一下 README.md」，模型调用 Read → 获得内容 → 生成回复 → 停止。全程用户只输入一次
- AC2: 输入「先找到所有 .py 文件，再搜索其中包含 import 的行」，模型可能先调 Glob → 拿到列表 → 再调 Grep → 汇总结果 → 回复。全程无需用户催促
- AC3: 将迭代上限设为 2，输入需要多轮工具的复杂任务 → Agent Loop 在第 2 轮后停止 → 终端显示停止原因为达到迭代上限
- AC4: 输入「你好」，模型不调用任何工具 → 直接生成文本回复 → Agent Loop 结束（0 轮工具调用）
- AC5: 输入「读一下 notexist.txt」，模型调用 Read → 返回文件不存在 → 模型基于错误信息调整（如搜索相似文件名）→ 不再盲目重试同一不存在的文件
- AC6: 输入「同时读 README.md 和 CLAUDE.md」，模型可能一次调用两个 Read → 两个 Read 并发执行 → 两个结果都返回给模型
- AC7: 输入「创建 a.txt 写入 hello，再创建 b.txt 写入 world」，模型调用两个 Write → 串行执行（a 先、b 后）→ 两个文件都被创建
- AC8: 模型一次请求同时调用 Read 和 Write → Read 先并发跑完 → Write 再串行跑 → 全部结果回灌
- AC9: 输入 /plan → 再输入「修改 README.md 的标题」→ 模型只能看到 Read/Glob/Grep → 无法调用 Write/Edit/Bash
- AC10: /plan 后输入 /do → 再输入「修改 README.md 的标题」→ 模型可使用全部六个工具 → 正常执行 Edit
- AC11: Agent Loop 执行中按 Esc → 当前轮次停止 → 回到输入等待状态，历史保留
- AC12: 注册表缺少工具时，模型连续 3 次尝试调用同一未注册工具 → Agent Loop 停止并提示
- AC13: 模拟网络中断 → 流错误触发停止 → 程序不崩溃 → 回到输入状态
- AC14: 一轮 Agent Loop 完成后，终端可见本轮消耗的 Token 数
- AC15: Agent Loop 执行中，终端可见当前轮次（如「第 1/25 轮」）
- AC16: 不配置工具时启动 PoorCode → 输入「你好」→ 流式回复正常 → 无 Agent Loop 相关错误
- AC17: 修改 config.yaml 中迭代上限为 5 → 启动 PoorCode → Agent Loop 在第 5 轮工具调用后停止
